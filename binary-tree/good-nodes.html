Count Good Nodes in Binary Tree

Step-by-step trace of goodNodes — watch maximumSoFar travel down each path and see which nodes clear the bar.

Pattern
🎯 Pre-order DFS + Path Maximum Propagation
Pass the running maximum down each path. At every node, check curr.val ≥ maxSoFar. Update the max before recursing into children.
Same pattern unlocks: #112 Path Sum · #113 Path Sum II · #543 Diameter · #110 Balanced Tree
📋 Problem Statement

Given a binary tree, count the number of good nodes. A node X is good if on the path from the root to X there are no nodes with a value greater than X's value. In other words — X must be the largest (or tied for largest) value seen so far on its root-to-node path.

⚠️ The root is always a good node — there are no ancestors to beat. A node does NOT need to be the maximum across the entire tree, only across its own root-to-node path.

Tree [3, 1, 4, 3, null, 1, 5] 4 good nodes
Tree [3, 3, null, 4, 2] 3 good nodes
Single node [1] 1 good node
💡 Intuition + Hints — Think Before You Code
1
What does "good" mean? A node is good if NO ancestor on the path from root is strictly greater. Equivalently: the node's value ≥ the maximum value seen so far on the path from root to here.
2
Pass the bar down. As you recurse, carry the running maximum. At each node, the "bar" it must beat is max(all ancestors). Update it with currMax = Math.max(maxSoFar, curr.val) before going deeper.
3
Use pre-order. Check the node first (root → left → right), then pass the updated max to children. You need to decide if a node is good BEFORE recursing into its subtree.
4
Seed with -Infinity. The root has no ancestors, so it always passes. Using -Infinity as the initial max makes this work automatically — any number is ≥ -Infinity. No special root case needed.
5
Closure counter. Keep goodNodeCount in the outer function. The inner traversal just increments it. No return value needed from the recursion — it's a side-effect accumulator.

Mental model: Think of it as a "high-score" check. As you walk from root toward a leaf, you're tracking the all-time high score along that specific path. A node is "good" if it beats (or ties) the current high score. When you turn left or right, your child inherits the updated high score — not the global tree max, just the max on their particular path from root.

⚡ The two-step per node: (1) Check: curr.val ≥ maxSoFar → if yes, count this node. (2) Update: currMax = Math.max(maxSoFar, curr.val) → pass to children. Every node does exactly these two things. The order matters — check first, then update the bar for kids.
goodNodes.js
var goodNodes = function(root) {
    let goodNodeCount = 0;

    function traversal(curr, maximumSoFar) {
        if (curr.val >= maximumSoFar) {    // Is this node ≥ all ancestors?
            goodNodeCount += 1             // Yes → it's a good node!
        }
        let currMax = Math.max(maximumSoFar, curr.val)   // Update bar for children
        curr.left  && traversal(curr.left,  currMax)
        curr.right && traversal(curr.right, currMax)
    }

    traversal(root, -Infinity)    // Root always passes (-Inf ≤ anything)
    return goodNodeCount
};

The bar only rises. currMax = Math.max(maxSoFar, curr.val) — the maximum along a path can only increase as you go deeper. Once a large value is seen, smaller descendants can't clear that bar.
▸
Pre-order is natural here. Check the node first (is it good?), then update the bar, then recurse. If you checked after recursing, the bar update would apply to the current node too — wrong.
▸

▸
Paths are independent. Going left vs going right gives children different inherited maxes. The path from root to a left leaf and the path to a right leaf are completely separate — each has its own running bar.

⚠️Edge Cases
rootThe root is always a good node — it has no ancestors, so it trivially beats all of them. The -Infinity seed ensures root.val ≥ -Infinity is always true.
singleSingle node tree → exactly 1 good node (the root itself). goodNodeCount = 1. Works naturally with no special handling.
tiedEqual values count as good: curr.val ≥ maxSoFar uses ≥, not >. A node whose value equals the maximum so far IS a good node.
all badStrictly decreasing path (e.g. 5→4→3→2→1 down one side) — only the first node on that path is good. Once a high value is seen, every smaller descendant is blocked.
all goodStrictly increasing path (e.g. 1→2→3→4→5) — every single node is good. Each clears the bar set by its predecessor. Result = total node count.
✦ Clean solution for revision
O(n) timeO(h) spacePre-order DFSClosure counter
var goodNodes = function(root) {
    let count = 0;
    function dfs(node, max) {
        if (node.val >= max) count++;          // ① check: is this node good?
        let newMax = Math.max(max, node.val);   // ② update: raise the bar
        node.left  && dfs(node.left,  newMax);  // ③ recurse with updated bar
        node.right && dfs(node.right, newMax);
    }
    dfs(root, -Infinity);   // seed: root always passes
    return count;
};
// Pattern: Pre-order DFS + propagate running max down
// A node is good iff its value ≥ max value on path from root to it
🚨Gotchas & Common Pitfalls
▸
Global Max instead of Path Max: A fatal mistake is making maxSoFar a global variable that updates for every node seen. The maximum is strictly path-dependent! It must be passed down recursively so left and right branches don't cross-contaminate.
▸
Updating bar before checking: If you write maxSoFar = Math.max(maxSoFar, curr.val) and THEN check curr.val >= maxSoFar... it will always be true! You must check if the node passes the old bar first.
⚖️Trade-offs & Alternatives
▸
Pre-order vs Post-order: For this specific problem, pre-order is naturally required. We MUST know the max from our ancestors before we can process the current node. Post-order doesn't make sense since children don't influence if the parent is "good".
▸
Closure Counter vs Returning Count: Our code uses an outer scope variable count++. Alternatively, the helper could return the count: return (isGood?1:0) + dfs(L) + dfs(R). Returning values is "purer", but the closure counter is slightly easier to trace and write in JS.
ELI5: Imagine walking down a mountain trail. You only stop to take a photo at a viewpoint if it's the highest altitude you've been to so far on this specific trail. You keep track of your personal "highest altitude seen," and take a photo every time you tie or break that record!
