var lowestCommonAncestor = function(root, p, q) {
  let lca = null

  function postOrderTraversal(curr) {
    if (!curr) return 0                         // null → contributes 0
    let count = 0
    let ansOnLeft  = postOrderTraversal(curr.left)
    let ansOnRight = postOrderTraversal(curr.right)

    if (curr.val == p.val || curr.val == q.val) {
      count++                                   // curr is p or q → count it
    }

    count = ansOnLeft + ansOnRight + count;
    if (count == 2 && !lca) {               // first time count==2 → LCA found!
      lca = curr
    }
    return count
  }

  postOrderTraversal(root);
  return lca
};

📋What the problem is asking

Given a binary tree and two nodes p and q, find their Lowest Common Ancestor (LCA) — the deepest node that has both p and q as descendants (a node is considered a descendant of itself).

In other words: go up from both p and q. The first node where their paths meet is the LCA. It's the deepest node that "covers" both targets.

💡Core intuition

Each node can count how many of {p, q} exist in its subtree (0, 1, or 2). This count bubbles bottom-up via post-order DFS.

The first node where count reaches 2 is the LCA — because that's the deepest node whose subtree contains both targets. Every ancestor above it will also have count=2, but they're higher up, so we use the !lca guard to capture only the first (deepest) one.

🧩Hints to reach the solution
  • 1Think about what each node should return to its parent: the count of p and q found in its subtree (0, 1, or 2).
  • 2At each node: count = ansOnLeft + ansOnRight + (curr is p or q ? 1 : 0).
  • 3When count === 2 AND lca is still null → set lca = curr. The !lca guard is critical to capture the deepest node, not a higher ancestor.
  • 4Use a closure variable let lca = null. The helper returns a number (count), while lca stores the result separately.
⚡Key trick / pattern

The !lca guard: Without it, every ancestor of the true LCA would also claim to be the LCA (all have count=2 by the time we reach them). The check if (count === 2 && !lca) ensures we lock in the first node that reaches count=2 — which post-order guarantees is the deepest.

Post-order is mandatory: We need children's counts before computing the parent's count. Pre-order or in-order wouldn't work — the information flows upward, not downward.

A node counts itself: If curr.val === p.val, count gets +1. This handles the case where p is an ancestor of q (p would be the LCA).

⚠️Edge cases to watch out for
ancestorp is ancestor of q (or vice versa): p=5, q=4, and 4 lives under 5. The LCA is 5 itself. When the algorithm visits node 5: left returns 0, right subtree returns 1 (found q=4), and curr matches p → count = 0+1+1 = 2 → LCA = node 5 ✓
rootLCA is the root: p=5 (left), q=1 (right). The root is the first node whose subtree contains both. count at root = leftCount(2) + rightCount(1) + 0 = 2 → LCA = root ✓
same nodep === q: A node is an ancestor of itself. The problem guarantees p ≠ q for this variant, but if they were equal, count would hit 2 only at an ancestor — that ancestor would be the LCA.
guaranteeBoth nodes exist: The problem guarantees both p and q exist in the tree. The count will always reach 2 somewhere, so lca will always be set before we return.
✦ Best solution for learning

var lowestCommonAncestor = function(root, p, q) {
  let lca = null;

  function postOrder(curr) {
    if (!curr) return 0;                        // null contributes 0

    let left  = postOrder(curr.left);             // count from left subtree
    let right = postOrder(curr.right);            // count from right subtree
    let self  = (curr.val == p.val || curr.val == q.val) ? 1 : 0;

    let count = left + right + self;

    if (count === 2 && !lca) lca = curr;       // first node with both targets

    return count;
  }

  postOrder(root);
  return lca;
};

📬

Count = signal

Each node returns how many of {p, q} are in its subtree (0, 1, or 2). This count travels upward like a message being passed to parents.

🎯

count == 2 → LCA

When count first reaches 2, both targets are in this node's world. The !lca guard ensures we capture the deepest such node, not a higher ancestor.

🔁

Post-order is crucial

Left → Right → Root. We must process children first to know their counts before deciding on the current node. Pre-order or in-order wouldn't work here.

🛡️

The !lca guard

Without it, every ancestor of the LCA would also claim to be the LCA (they all eventually see count=2). The guard says: "first one wins — that's the deepest."

⚠️ Gotchas & Common Pitfalls
  • Locking the LCA: You must use a guard (e.g. !lca) to capture the first node that reaches count 2. Otherwise, every ancestor node above it will also see count 2 and overwrite your LCA.
  • Traversal Order: Pre-order won't work! You must use post-order (children then parent), because a parent's count strictly relies on gathering its children's counts first.
  • Self Matching: Don't forget to count the current node if curr.val === p.val. p can be the LCA of q.
⚖️ Trade-offs & Alternatives
  • Post-Order Check (Current): O(N) Time, O(H) Space. Best approach, elegant and generic for any binary tree.
  • Store Paths: Alternative is to traverse the tree, store the paths from root to p and root to q in arrays, then compare arrays iteratively. Wastes O(N) extra space and requires 2+ passes.
Time: O(n) — each node visited once
Space: O(h) — call stack depth = height
Traversal: post-order DFS
ELI5: Each node tells its parent "I found X targets in my family." The very first node to hear "I found 2!" from its family shouts "I'M THE LCA!" and locks it in.
