📋 Problem Statement

Given a binary tree, return its level-order traversal in a zigzag pattern: level 0 goes left→right, level 1 goes right→left, level 2 goes left→right again — alternating every level.

Output is a list of lists. Each inner list contains all node values at that level in the correct zigzag direction.

Tree [3, 9, 20, null, null, 15, 7] [[3],[20,9],[15,7]]
Tree [1, 2, 3, 4, 5, 6, 7] [[1],[3,2],[4,5,6,7]]
💡 Intuition + Hints

2

3
The entire zigzag trick: levelArr.push(val) for even levels → appends to end → left-to-right order. levelArr.unshift(val) for odd levels → inserts at front, building the array in reverse → right-to-left order. No extra reversal step needed.
4
After the inner loop: ans.push(levelArr) and level++. The len snapshot is critical — it prevents accidentally processing next level's nodes in this loop.

Core insight: The queue always processes nodes left to right. The zigzag comes entirely from where you insert into the result array, not from how you traverse. unshift() prepends every new value, so by the time a level is done, the array is naturally reversed.

zigzagLevelOrder.js 
var zigzagLevelOrder = function(root) {
    if (!root) return []
    let level = 0;
    let queue = [root]
    let ans = []

    while (queue.length) {
        let len = queue.length;   // snapshot: nodes at this level
        let levelArr = []
        for (let i = 0; i < len; i++) {
            let curr = queue.shift();   // dequeue front
            if (level % 2 !== 0) {
                levelArr.unshift(curr.val)    // odd level → insert at front
            } else {
                levelArr.push(curr.val)       // even level → append to end
            }
            if (curr.left)  queue.push(curr.left)
            if (curr.right) queue.push(curr.right)
        }
        ans.push(levelArr)
        level++
    }
    return ans;
};

📋What the problem is asking

Given a binary tree, return its level order traversal but in a zigzag pattern — level 0 goes left→right, level 1 goes right→left, level 2 goes left→right again, and so on alternating.

Output is a list of lists: each inner list holds the node values at that level, already in the correct zigzag direction.

💡Core intuition

Standard BFS gives nodes left→right at every level. For zigzag, we don't change how we process nodes — we change how we insert their values into the level array.

Even levels: push(val) → appends to end → L→R order.
Odd levels: unshift(val) → inserts at front → R→L order. No reversals needed!

🧩Hints to reach the solution
  • 1Start with plain BFS. Use a queue, snapshot len = queue.length before each level's inner loop.
  • 2Track a level counter starting at 0. Check level % 2 to decide insert direction.
  • 3push() for even levels (0, 2, 4…) and unshift() for odd levels (1, 3, 5…). That's the entire zigzag trick.
  • 4After the inner loop finishes, push levelArr into ans and increment level.
⚡Key trick / pattern

push vs unshift: Both process nodes in the same BFS order (left→right). But unshift() inserts each new value at index 0, pushing previous values right. After processing a full level with unshift(), the array ends up in reverse order — exactly right→left.

The len snapshot: let len = queue.length before the inner loop is critical. As you process this level's nodes, you're also enqueuing next level's children. Without snapshotting len, you'd accidentally process the next level's nodes too.

Why not reverse? You could build L→R then reverse for odd levels — same result, but O(n) extra work per level. unshift() is cleaner.

⚠️Edge cases to watch out for
nullEmpty tree: Return [] immediately. Without this guard the queue starts as [null] and the loop runs once trying to access null.left — crash.
singleSingle node: Level 0, even → push(root.val) → [[root.val]]. Works naturally.
level countLevel counter vs boolean: level % 2 and a toggled boolean leftToRight both work equally well. The key is flipping direction after every level. Pick one and stay consistent.
noteunshift is O(n): Each unshift() call shifts all existing elements. For very large levels this is slower than using a deque or building reversed. For interviews, unshift() is acceptable and cleaner.
✦ Best solution for learning
O(n) time O(n) space BFS + level tracking push / unshift trick 
var zigzagLevelOrder = function(root) {
  if (!root) return [];              // edge case: empty tree
  let queue = [root], ans = [], level = 0;

  while (queue.length) {
    let len = queue.length;          // snapshot: # nodes at this level
    let levelArr = [];

    for (let i = 0; i < len; i++) {
      let curr = queue.shift();      // dequeue front node
      if (curr.left)  queue.push(curr.left);
      if (curr.right) queue.push(curr.right);

      if (level % 2 === 0) levelArr.push(curr.val);    // even: L→R
      else               levelArr.unshift(curr.val); // odd:  R→L
    }

    ans.push(levelArr);
    level++;
  }
  return ans;
};

// Key insight: BFS traversal order never changes (always L→R).
// Only the INSERTION into levelArr changes: push() vs unshift().
// unshift() builds the array in reverse by prepending each new value.
// len snapshot prevents processing next level's nodes in the inner loop.

🔄

BFS with a Queue

A queue processes nodes level by level. We capture len = queue.length at the start of each level so we know exactly how many nodes belong to this level — without mixing up the next level's nodes.

📍

push() vs unshift()

On even levels: levelArr.push(val) appends to the end → left-to-right order. On odd levels: levelArr.unshift(val) inserts at the front → right-to-left order. No array reversal needed!

📸

len snapshot trick

We save len = queue.length before the inner loop. As we process nodes, we enqueue their children — but the loop only runs len times, so only this level's nodes are processed.

🎯

Why level % 2 works

Even levels (0, 2, 4...) use push — natural order. Odd levels (1, 3, 5...) use unshift — reversed. Together they create the zigzag snake pattern across the tree.

Time: O(n) — every node visited once
Space: O(n) — queue + output storage
Pattern: BFS with alternating insert
Key ops: push / unshift / shift
