Pattern
Two Pointers — Read / Write
The array is sorted, so duplicates are always adjacent. One pointer (k) marks the last unique write position; a second pointer (i) scans ahead. When arr[i] > arr[k], a new unique value is found — advance k and write. Zero extra space needed.
#80 Remove Duplicates II #283 Move Zeroes #27 Remove Element
2Problem Statement

Given a sorted integer array nums, remove duplicates in-place so each unique element appears only once. Return k — the count of unique elements. The first k positions of the array must hold those unique values.

Think of it like compressing a sorted list of names on a whiteboard: you keep one pointer at the last confirmed unique name, and scan ahead — whenever you hit a new name, overwrite the next slot.
Example 1
arr = [1,1,2]
k = 2, arr = [1,2,_]
Example 2
arr = [0,0,1,1,1,2,2,3,3,4]
k = 5, arr = [0,1,2,3,4,...]
Edge case
arr = [1]
k = 1, arr = [1]
3Intuition & Hints
1 Brute force: Copy to a Set, convert back — O(n) time, O(n) space. Works, but the constraint says "in-place."
2 Key insight: Because the array is sorted, duplicates are always neighbours. You don't need to search — just compare arr[i] with the last written value at arr[k].
3 Two pointers: k = 0 anchors the write head. i runs from 1 to end. Whenever arr[i] > arr[k], we found a new unique: do k++ then arr[k] = arr[i].
4 Return value: k is a 0-based index of the last unique slot. Count = k + 1.
Key trick: arr[i] > arr[k] (strict greater-than) works because the array is sorted — a larger value guarantees a new unique element.
4Solution Code
JavaScript 
/**
 * @param {number[]} nums
 * @return {number}
 */
var removeDuplicates = function(arr) {
    let k = 0;
    for (let i = 1; i < arr.length; i++) {
        if(arr[i] > arr[k]) {
            k++;
            arr[k] = arr[i]
        }
    }
    return k + 1;
};

6Gotchas & Common Pitfalls
① Off-by-one in the return value
k is a 0-based index, so the count of unique elements is k + 1. Returning just k will be wrong for every case with at least one element. LeetCode checks expectedNums[k-1], so it will silently truncate the last unique element.
② Starting i at 0 instead of 1
If you start i = 0, you compare arr[0] with arr[0] — always equal, always skipped. The loop then misses comparing the first two elements. Always start the scan pointer one ahead of the write pointer.
③ Using >= instead of >
Because the array is sorted (not just distinct), arr[i] > arr[k] is correct. Using >= would re-write values that are the same — no correctness issue here, but it's conceptually wrong and would break if you tried to adapt this to "allow at most 2 duplicates" (LeetCode #80).
7Trade-offs & Alternatives
Why not use a Set or filter()?
Both produce a new array — O(n) extra space. The problem requires in-place modification. Also, filter would need something to compare against, effectively re-implementing the two-pointer idea anyway.
Could we use splice() instead of overwriting?
splice to remove elements inside a loop is O(n²) because each splice shifts all subsequent elements. The write-pointer approach is O(n) — far superior.
Space trade-off
This solution is O(1) extra space. The trade-off is that the "tail" of the original array (positions k+1 onwards) is left with stale data — but the problem explicitly allows this since only the first k elements are evaluated.
