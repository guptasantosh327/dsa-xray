Two Sum II — Input Array Is Sorted

Easy Two Pointers Sorted Array O(n) Time O(1) Space
Pattern
Two Pointers — Converging Inward
The array is sorted — that's the key constraint that makes this pattern work. Instead of scanning every pair O(n²), place one pointer at the smallest element and one at the largest. Their sum tells you exactly which direction to move: too big → shrink right, too small → grow left. The sorted order guarantees you'll never miss the answer.
#1 Two Sum (HashMap) #15 3Sum #125 Valid Palindrome
Problem

Given a 1-indexed sorted array numbers, find two numbers that add to target. Return their indices as [index1, index2] (1-based). Exactly one solution is guaranteed.

Analogy: Imagine a number line on a ruler. You place your left thumb at the smallest number and your right thumb at the largest. Check their sum. If it's too big, slide your right thumb left. Too small — slide your left thumb right. The sorted order means this always converges on the answer.

Example 1
numbers = [2, 7, 11, 15], target = 9 → [1, 2]   // 2+7=9
Example 2
numbers = [2, 3, 4], target = 6 → [1, 3]   // 2+4=6
Example 3 — multiple moves needed
numbers = [1, 3, 4, 5, 7, 11], target = 9 → [3, 4]   // 4+5=9
Intuition & Hints
  • Brute force: Try every pair (i, j) with i < j. O(n²) time — works but ignores the sorted constraint entirely.
  • Binary search: Fix each element, binary-search for the complement. O(n log n). Better, but still not fully exploiting sorted order.
  • Two pointers (optimal): Start i=0 (smallest), j=n-1 (largest). Their sum is a compass. Too big → the largest number is too large, so j--. Too small → the smallest number is too small, so i++. O(n), O(1).
  • Why this is safe: When sum > target, every element to the left of j is ≤ numbers[j], so any pair with j would still be too big. Moving j left is the only useful action. Same logic applies in reverse for i.
Key Insight The sorted property turns an O(n²) search space into a directed walk. Each step eliminates an entire row or column of pairs — you never need to revisit them.
Your Code
var twoSum = function(numbers, target) {
    let i = 0;
    let j = numbers.length - 1;
    while ( i < j) {
        let sum = numbers[i] + numbers[j]
        if( sum > target) {
            j--
        } else if( sum < target) {
            i++
        } else {
            return [i +1, j +1]
        }
    }
};

Gotchas & Common Pitfalls
Returning 0-indexed instead of 1-indexed

The problem explicitly says 1-indexed. The most common mistake is returning [i, j] instead of [i+1, j+1]. Always re-read the output format.

Using a HashMap (like Two Sum #1)

Instinct says "HashMap" because it's Two Sum. But #167 adds the sorted constraint and O(1) space requirement — the two-pointer approach is strictly better here.

The while condition i < j, not i <= j

When i === j, both pointers point to the same element. You'd be counting a single number twice. The problem guarantees distinct indices, so stop when they meet.

Trade-offs & Alternatives
Two Pointers (used)

O(n) time, O(1) space. Optimal. Works because the array is sorted — the sorted property is load-bearing. This approach would be invalid on an unsorted array.

HashMap (not used)

O(n) time, O(n) space. Valid but wasteful — ignores the sorted constraint, spends extra memory on a lookup table that the ordering already provides.

Binary Search per element

O(n log n) time, O(1) space. Fix each element, binary-search for complement. Correct and space-efficient, but the two-pointer approach is asymptotically faster.

Brute Force

O(n²) time, O(1) space. Try every pair. Guaranteed to find the answer but completely ignores the problem's hints — never acceptable in an interview.
