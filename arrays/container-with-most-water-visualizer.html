#11

Container With Most Water

Medium Two Pointers Greedy O(n) Time O(1) Space
Pattern
Two Pointers — Shrink From Wider
The widest container is always the starting candidate — it maximises the width factor. From there, the only hope of beating it is finding a taller pair. The shorter wall is the bottleneck; moving the taller wall inward can only make things worse (same height limit, less width). So always discard the shorter wall. This greedy choice is safe — you never skip a potentially better answer.
#42 Trapping Rain Water #167 Two Sum II #15 3Sum
Problem

Given an integer array height of length n, there are n vertical lines where the i-th line has height height[i]. Find two lines that, together with the x-axis, form a container that holds the most water. Return the maximum amount of water.

Analogy: Imagine a row of fence posts of different heights. You're choosing two posts to stretch a tarp between them. The water held = the shorter post's height × the distance between them. Start with the widest possible span. You can only beat it by finding a taller shorter-post — so always replace the shorter post first.

Example 1 — classic case
height = [1,8,6,2,5,4,8,3,7] → 49  // lines at i=1 (h=8) and j=8 (h=7): min(8,7)×7 = 49
Example 2 — only two lines
height = [1, 1] → 1  // min(1,1)×1 = 1
Example 3 — tallest walls not the answer
height = [4, 3, 2, 1, 4] → 16  // lines at i=0 (h=4) and j=4 (h=4): min(4,4)×4 = 16
Intuition & Hints
  • Brute force: Try every pair (i, j). Area = min(h[i], h[j]) × (j–i). Track max. O(n²) — works, but ignores how area changes as pointers move.
  • Key observation: Area is constrained by two factors: width and the shorter wall. Start at maximum width (i=0, j=n-1). Moving inward always reduces width — the only way to compensate is with a taller short wall.
  • Greedy move: If height[i] < height[j], moving j inward can't help — the area is already limited by the left wall. But moving i inward might find a taller wall. So always move the shorter pointer.
  • Why greedy is safe: When you discard pointer i, you're discarding all pairs (i, j-1), (i, j-2)… — every one of them has smaller width AND is still bounded by height[i]. None can beat the current area.
Key Insight Moving the shorter wall is the only move that keeps hope alive. Moving the taller wall guarantees a worse or equal outcome (less width, same height limit).
Your Code
var maxArea = function (height) {
    let i = 0;
    let j = height.length - 1
    let max = 0
    while (i < j) {
        let h = Math.min(height[i], height[j])
        let area = h * (j - i);
        max = Math.max(area, max)
        if (height[i] < height[j]) {
            i++
        } else {
            j--
        }
    }
    return max
};

Gotchas & Common Pitfalls
Moving the taller wall, not the shorter one

The most common mistake. If you move the taller wall inward, you get less width AND the bottleneck height stays the same (still limited by the shorter wall). You can only gain by moving the shorter wall — it's the only variable that can improve.

Forgetting equal heights

When height[i] === height[j], the code falls into the else branch and decrements j. That's fine — either pointer can be moved when they're equal (both sides are the bottleneck). Some solutions handle this as a third branch; your solution's else is correct.

Confusing area formula: it's min height × width, not sum

Water fills to the brim of the shorter wall. The area is Math.min(height[i], height[j]) × (j - i). A common slip is using height[i] * height[j] or adding heights instead of multiplying by width.

Trade-offs & Alternatives
Two Pointers (used)

O(n) time, O(1) space. Optimal. The greedy insight — always move the shorter pointer — is the key that makes this work. Provably correct and a classic interview pattern.

Brute Force (not used)

O(n²) time, O(1) space. Try every pair. Correct but unacceptable at scale — with n=100,000 lines, that's 5 billion pairs. Always rejected in interviews.

Why not Divide & Conquer?

The optimal pair can straddle the midpoint, making divide-and-conquer harder to implement correctly and only getting to O(n log n). No benefit over the linear two-pointer pass.

Note: unlike #42 Trapping Rain Water

#42 fills every gap, requiring prefix/suffix max arrays or a more involved two-pointer. This problem is simpler: one container, two walls, greedy one-pass suffices.
