Sort intervals by start, then greedily merge each one into the running tail of the result — no need to look back further than one step.

Problem Statement — #56 Merge Intervals

Given an array of intervals where intervals[i] = [starti, endi], merge all overlapping intervals and return an array of the non-overlapping intervals that cover all the input intervals.

Think of it like a busy schedule: if a meeting runs 9–11am and another starts at 10am and ends at 12pm, you're blocked 9am–12pm. The goal is to collapse all your overlapping blocks into clean, continuous busy windows.
Standard overlap
[[1,3],[2,6],[8,10],[15,18]]
→ [[1,6],[8,10],[15,18]]
Touching intervals
[[1,4],[4,5]]
→ [[1,5]]
Contained interval
[[1,10],[2,3],[4,6]]
→ [[1,10]]
3
Intuition + Hints
  • 1 Brute force O(n²): Compare every pair of intervals. Too slow. Can we reduce comparisons?
  • 2 Key insight: If intervals are sorted by start, a new interval can only overlap with the last merged interval. No need to scan backwards — one comparison suffices.
  • 3 Optimal O(n log n): Sort once, then do one linear pass. Track the tail of the result with lastIndex to write back in-place into ans.
★ Key trick: The overlap check is ans[lastIndex][1] >= arr[i][0]. When true, take Math.max of the two ends — this handles the "contained interval" case for free (where B is inside A, we keep A's end).

Solution Code
merge.js 
1var merge = function(intervals) {


2    const arr = intervals.sort((a, b) => a[0] - b[0])


3    let lastIndex = 0;


4    let ans = [arr[0]]


5    for(let i = 1; i < arr.length; i++){


6        if(ans[lastIndex][1] >= arr[i][0]) {


7            ans[lastIndex][1] = Math.max(ans[lastIndex][1], arr[i][1])


8        } else {


9            lastIndex++;


10            ans[lastIndex] = arr[i]


11        }


12    }


13    return ans


14};

Gotchas & Common Pitfalls
  • ⚠
    Forgetting Math.max on the end. If interval B is fully contained inside interval A (e.g. A=[1,10], B=[2,3]), naively assigning arr[i][1] would shrink the merged interval. Always use Math.max(ans[lastIndex][1], arr[i][1]).
  • ⚠
    Using > instead of >= for the overlap check. Intervals that share an endpoint (e.g. [1,4] and [4,5]) do overlap. The condition must be ans[lastIndex][1] >= arr[i][0] to catch the "touching" case.
  • ⚠
    Mutating the input. Array.prototype.sort() sorts in-place. If the caller relies on the original order, clone first: [...intervals].sort(...). Always ask in an interview whether mutation is acceptable.

Trade-offs / Alternatives
Why not a stack? A stack-based solution works and is idiomatic, but it's equivalent — you push/pop based on the same overlap check. Using an ans array with a lastIndex pointer achieves the same O(1) tail access without any extra data structure overhead.
Why can't we do better than O(n log n)? The sort dominates. The merge pass itself is O(n). There is no known O(n) algorithm for unsorted input — you fundamentally need the ordering to guarantee that overlapping intervals are adjacent.
True in-place? This solution builds a new ans array, so space is O(n) for the output. You can overwrite the input array in-place (pointer into the same array), reducing auxiliary space to O(1) excluding output — but it makes the code harder to read and is rarely asked.

ELI5 Mnemonic
"Sort the meetings by start time. Walk through them — if the next starts before the current block ends, extend the block; otherwise, start a new block."
Sort first — makes it so you only ever need to compare with the last result, turning O(n²) into O(n).
Math.max on the end — the one-liner that handles both "extending" and "contained" cases simultaneously.
lastIndex pointer — avoids a separate stack; write directly into ans without extra allocations.
