#49 · Group Anagrams

Pattern
Frequency-Array as HashMap Key
The core insight: two strings are anagrams if and only if they have identical character frequency profiles. Instead of sorting (O(k log k) per string), we encode each string as a 26-slot count array and serialize it into a canonical string key — giving O(k) grouping and a guaranteed collision-free fingerprint for any anagram group.
#242 Valid Anagram #438 Find All Anagrams in a String #266 Palindrome Permutation
Problem

Given an array of strings strs, group the anagrams together. You can return the answer in any order. An anagram uses the same letters in the same quantities, just rearranged.

Analogy: Imagine you're sorting a pile of Scrabble tiles that someone scattered randomly. Two bags of tiles are "the same" if they contain exactly the same letters in the same quantities. Your job is to sort all the bags into matching groups — without caring about the order of tiles inside each bag.

Example 1
["eat","tea","tan","ate","nat","bat"]
→ [["bat"],["nat","tan"],["ate","eat","tea"]]
Example 2
[""]
→ [[""]]
Example 3
["a","b","a"]
→ [["a","a"],["b"]]
Intuition & Hints
  • Brute Force — Sort Each String: Sort every string alphabetically. Anagrams will produce identical sorted strings. Use the sorted string as the map key. Works, but sorting takes O(k log k) per string. Total: O(n·k·log k).
  • Optimal — Frequency Array Key: Instead of sorting, count the 26 letter frequencies in O(k). Encode the array as "#1#0#0#0#1#..." using '#' separators so counts ≥ 10 don't collide (e.g. "#10#0" ≠ "#1#00"). Total: O(n·k).
  • Group by Key: Use a plain object as the map. For each string, compute its key; if key exists, push to that array; otherwise create a new array. Return Object.values(keyMap).
  • Why '#' separator matters: Without it, hashArr = [10,0,...] would stringify to "100..." which collides with [1,0,0,...] → "100...". The '#' prefix makes each slot unambiguous.
✦ Key insight: charCodeAt(0) - 'a'.charCodeAt(0) maps any lowercase letter to its 0-based index in the alphabet. 'a'→0, 'b'→1, ..., 'z'→25. This turns a character lookup into a direct array index — no hash function needed.
Solution Code
javascript
your solution — exact 
var groupAnagrams = function(strs) {
    let keyMap = {}
    // Create a array which will have all the counts of the character
    for(let i = 0; i < strs.length; i++) {
        let s = strs[i]
        let hashArr = new Array(26).fill(0)
        for(let j = 0; j < s.length; j++) {
            let index = s[j].charCodeAt(0) - 'a'.charCodeAt(0)
            hashArr[index] += 1
        }
        let key = ''
        for(let z = 0; z < hashArr.length; z++) {
            key += '#' + hashArr[z]
        }
        if(!keyMap[key]){
            keyMap[key] = [s]
        } else {
            keyMap[key].push(s)
        }
    }
    return Object.values(keyMap)
};

Gotchas & Common Pitfalls
The '#' separator is not optional. Without it, hashArr = [1,0,0,...,10,...] would produce key "100...10...", which collides with [10,0,...,1,0,...] → "100...10...". Using '#' + count makes each slot unambiguous: "#1#0#0...#10..." vs "#10#0...#1#0...".
Empty strings are valid input and should group together. An empty string produces hashArr all-zeros, key "#0#0...#0". Multiple empty strings will correctly map to the same key and form one group — don't add a length check that skips them.
Don't sort; use frequency arrays. Many candidates reach for s.split('').sort().join('') as the key. It works but costs O(k log k) per string. The freq-array key is O(k) and often catches interviewers' attention as the optimal choice.
Trade-offs & Alternatives
Sorted string key vs frequency array key: s.split('').sort().join('') is simpler to write and read but is O(k log k) per string. The frequency array key is O(k) — better when strings are long or n is large. In an interview, start with sorted key, then offer the frequency key as an optimization.
Object vs Map: A plain JS object {} is fine here because keys are always non-symbol strings. A Map would have the same O(1) amortized operations but slightly more overhead. The object approach is idiomatic for this problem.
Space trade-off: We use O(n·k) extra space — the keyMap stores all strings. This is unavoidable since the output itself is O(n·k). The hashArr is reused per iteration so it's only O(26) = O(1) extra per loop iteration.
