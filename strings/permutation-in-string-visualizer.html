Fixed-Size Sliding Window + Frequency Array
A permutation has the same characters in the same counts — just rearranged. So instead of generating all permutations, maintain a 26-slot frequency array for the current window and compare it against s1's frequency array. Since the window size is fixed (= s1.length), sliding is O(1): remove the leftmost char, add the rightmost.
#438 Find All Anagrams in a String #76 Minimum Window Substring #3 Longest Substring Without Repeating
Problem Statement

Given two strings s1 and s2, return true if s2 contains a permutation of s1, or false otherwise. In other words, one of s1's permutations must be a contiguous substring of s2.

"Imagine s1 is a handful of scrabble tiles. Slide that exact same set of tiles along s2 one step at a time — if any window of s2 uses the exact same tiles (regardless of order), return true."
Example 1
s1 = "ab"
s2 = "eidbaooo"
✓ true — window "ao" matches {a:1,b:0}?... wait: "ba" at index 3-4
Example 2
s1 = "ab"
s2 = "eidboaoo"
✗ false — 'a' and 'b' never appear adjacent
Edge: single char
s1 = "a"
s2 = "a"
✓ true — window size 1, exact match
Intuition + Hints
  • 1Brute force: generate all permutations of s1, check if any is a substring of s2. O(k! · n) — catastrophically slow even for k=10.
  • 2Key insight: two strings are permutations of each other if and only if they have the exact same character frequency distribution.
  • 3So the question becomes: does any window of size s1.length in s2 have the same frequency map as s1? Fixed window → classic sliding window.
  • 4Initialize both frequency arrays for the first window, then slide: decrement the leaving char, increment the arriving char. O(1) per slide.
  • 5Compare the two 26-element arrays once per window position — that's O(26) = O(1). Total: O(n) time, O(1) space.
Key Trick: charCodeAt(k) - 'a'.charCodeAt(0) maps any lowercase letter to index [0, 25]. A 26-slot int array is a perfect, collision-free hash for lowercase ASCII strings.
Solution Code
Active Recall mode — blur hides core logic. Active line is highlighted during visualization. 
/**
 * @param {string} s1
 * @param {string} s2
 * @return {boolean}
 */
var checkInclusion = function(s1, s2) {
    let window = s1.length
    let hashW = new Array(26).fill(0)
    let hashM = new Array(26).fill(0)
    for (let k = 0; k < window; k++) {
        let indexM = s1.charCodeAt(k) - 'a'.charCodeAt(0)
        let indexW = s2.charCodeAt(k) - 'a'.charCodeAt(0)
        hashM[indexM] += 1
        hashW[indexW] += 1
    }
    let i = 0
    let j = s1.length - 1
    while (j < s2.length) {
        if (isSameHash(hashW, hashM)) {
            return true
        } else {
            hashW[s2.charCodeAt(i) - 'a'.charCodeAt(0)] -= 1
            i++
            j++
            hashW[s2.charCodeAt(j) - 'a'.charCodeAt(0)] += 1
        }
    }
    return false
}

function isSameHash(hash1, hash2) {
    let hashS1 = ''
    let hashS2 = ''
    for (let i = 0; i < 26; i++) {
        hashS1 += '#' + hash1[i]
        hashS2 += '#' + hash2[i]
    }
    return hashS1 == hashS2
}

Gotchas & Common Pitfalls
  • ⚠ Out-of-bounds slide: When j = s2.length - 1 and no match, the code does j++ then reads s2.charCodeAt(j) at an invalid index. In JavaScript this returns NaN, so hashW[NaN] += 1 writes to a named property and doesn't corrupt the array — but it's a latent bug. Safer: add a if (j + 1 < s2.length) guard before sliding.
  • ⚠ j initialization: j starts at s1.length - 1, not s1.length. The first window is [0, s1.length-1] — zero-indexed and fully populated before any comparison. Getting this wrong shifts every window by one.
  • ⚠ Shadowing the global window: The variable name window shadows the browser's global Window object. It works here because the global is never accessed, but in a larger codebase or interview context it's worth flagging. Consider winSize or k instead.
Trade-offs & Alternatives
Why not sort each window?
Sorting a window of size k costs O(k log k). Doing that for each of the O(n) windows = O(nk log k) total. Frequency array comparison is O(26) = O(1) per window → O(n) total. The frequency array wins decisively for larger alphabets too if you cap the alphabet size.
Stringification vs element-wise compare in isSameHash
The solution concatenates arrays into strings ('#' + hash[i]) and compares strings. This is technically O(26) string allocations per call. An element-wise for loop with early exit is faster in practice because it can short-circuit. Both are O(1) asymptotically since the alphabet is bounded.
Could use a "match counter" optimization
An alternative approach (common in #438) tracks how many of the 26 character counts currently match between the two arrays. Only update that counter when sliding. This reduces the per-window comparison from O(26) to O(1), though both are effectively constant.
