Search in a Binary Search Tree

Step-by-step trace of searchBST — watch how the BST ordering property lets us eliminate entire subtrees at each step, making search O(log n).

Pattern
🎯 BST Binary Search — Eliminate Half at Every Step
At every node: if target is smaller, go left (ignore right). If larger, go right (ignore left). The BST ordering makes this safe.
Same pattern unlocks: #701 Insert into BST · #98 Validate BST · #270 Closest BST Value
📋Problem Statement
BST Superpower: Because left subtree values are always smaller and right subtree values are always larger, you never need to search both sides. One comparison tells you exactly where to go — eliminating half the tree every step.

Given the root of a BST and an integer val, find the node in the BST where node.val === val and return the subtree rooted at that node. If no such node exists, return null.

search val=2 subtree [2,1,3]
search val=5 null — not found
💡Intuition + Hints
1
Base cases first. If root is null, the value doesn't exist — return null. If root.val === val, found it — return root (the whole subtree, not just the value).
2
Use the BST property. If root.val > val, the target can ONLY be in the left subtree — the right side is all bigger, so skip it entirely.
3
Mirror logic. If root.val < val, target can ONLY be in the right subtree. One comparison, one direction, no backtracking.
4
The ternary root.val > val ? searchBST(root.left, val) : searchBST(root.right, val) is the whole algorithm. Clean and elegant.
searchBST.js
var searchBST = function(root, val) {
  if(!root) return root                // ① null → not found
  if(root.val == val) return root       // ② exact match → return subtree

  return root.val > val
    ? searchBST(root.left, val)         // ③ curr too big → go LEFT
    : searchBST(root.right, val)        // ④ curr too small → go RIGHT
};

🧠Mental Model — Think of a Phone Book
▸
A BST is a sorted phone book. You don't scan every name — you open to the middle and immediately decide: go left (earlier in alphabet) or go right (later). Each check eliminates half the remaining entries.
▸
The ternary is everything. root.val > val ? go left : go right. One comparison, one recursive call, no backtracking, no branching. The BST property guarantees you'll never need the other side.
▸
Return the node, not the value. The problem asks for the subtree rooted at the found node. When you find the match, return root (the whole node), not root.val.
▸
Why return root for null? Writing if(!root) return root is the same as return null — but it's idiomatic JS for "return whatever root is". Both work identically.
🛑

Base case: null

Reached a null node — we fell off the tree without finding the value. The target doesn't exist. Return null.

✅

Base case: match

root.val === val — found it! Return the entire node (root), which is the subtree rooted there.

⬅️

Go left

When root.val > val: target is smaller, so it MUST be in the left subtree. The entire right side can be safely ignored.

➡️

Go right

When root.val < val: target is larger, so it MUST be in the right subtree. The entire left side can be safely ignored.

⚠️Edge Cases
emptyEmpty tree: root = null. The first if(!root) fires immediately and returns null. Handles it without any extra code.
not foundValue not in tree: You keep recursing until you hit null. The null base case returns null, which propagates back up through every return searchBST(...) call.
root is targetTarget is the root itself: root.val === val matches immediately on the first call. Returns the entire original tree — perfect, that's the subtree rooted at root.
complexityO(log n) vs O(n): Balanced BST → O(log n) because you halve the search space each step. Skewed/unbalanced BST (like a linked list) → O(n) worst case. Same algorithm, different tree shape.

var searchBST = function(root, val) {
  if(!root)              return root  // null → not found
  if(root.val == val)   return root  // found it → return subtree
  return root.val > val
    ? searchBST(root.left,  val)     // too big → go left
    : searchBST(root.right, val)     // too small → go right
};
