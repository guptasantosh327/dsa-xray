Kth Smallest in BST

Step-by-step trace of kthSmallest — watch in-order DFS visit nodes in ascending order. The kth node visited is the answer.

Pattern
🎯 In-order DFS + Counter + Early Exit
In-order traversal of any BST visits nodes in ascending sorted order. Decrement a counter at each root visit. When it hits zero, that node IS the kth smallest.
Same pattern unlocks: #538 Convert BST to Greater Tree · #99 Recover BST · #272 Closest BST Values II
📋 Problem Statement
BST Superpower for this problem: In-order traversal (left → root → right) of a BST visits every node in ascending sorted order. So the kth node visited is literally the kth smallest value — no sorting needed!

Given the root of a BST and an integer k, return the kth smallest value (1-indexed) among all node values.

BST [3,1,4,null,2], k=1 1st smallest = 1
BST [5,3,6,2,4,null,null,1], k=3 3rd smallest = 3
💡 Intuition + Hints
1
BST in-order = sorted array. In-order traversal visits: left subtree (all smaller) → current node → right subtree (all larger). This is the BST ordering property — it gives ascending order for free.
2
Don't collect everything. You only need the kth element. Use a count variable. Decrement it each time you "visit" a node (at the root step of in-order). When it reaches 0, you've found it.
3
Early exit saves work. Once you find the answer, stop all recursion. The if(ans != null) return guard short-circuits every remaining recursive call immediately.
4
Closure variables. Both ans and count live outside the recursive function — they persist across all recursive calls, acting as shared state.

Mental model: Imagine BST nodes laid out in a sorted line. In-order DFS visits them exactly left-to-right in that line. You're just counting steps along the line — when your step counter hits k, record the current node and stop walking.

⚡ The two-role trick: count counts DOWN from k. At each "root" step of in-order, decrement count. When count hits exactly 0, the current node is the kth. The if(ans != null) return guard then ensures ALL subsequent recursion is skipped — O(k + h) in the best case instead of O(n).
kthSmallest.js
var kthSmallest = function(root, k) {
  let ans = null
  let count = k            // counts down from k to 0

  const traversal = (curr) => {
    if (ans != null) return  // ← early exit: answer already found

    curr.left && traversal(curr.left)   // ① left subtree first

    --count;                             // ② visit this node
    if (count == 0) ans = curr.val      // kth visit = answer!

    curr.right && traversal(curr.right)  // ③ right subtree last
  }

  traversal(root)
  return ans
};

🧠Mental Model — The Sorted Line Walk
▸
In-order on BST = walking sorted values left-to-right. You don't need a sorted array. The BST IS the sorted structure — in-order traversal just unrolls it in order.
▸
Count down, not up. Start count at k. Decrement at each node visit. When count hits 0 — you're standing exactly on the kth node.
▸
Early exit = the real optimization. Standard in-order visits all n nodes. With the if(ans != null) return guard, we stop as soon as count hits 0. For small k, this is huge.
▸
Closure variables are the glue. ans and count live outside the recursive function. They're shared across all call frames — no need to pass them as parameters or thread them through return values.

⚠️Edge Cases
k=1Leftmost node is always the minimum. In-order visits it first. count goes from 1→0 immediately → returns with minimal traversal.
k=nRightmost node is the maximum. Must traverse the entire tree. The if(ans != null) return guard never fires until the very last node.
Single nodeOnly node, k must be 1. traversal visits it, count 1→0, ans set immediately. Returns correctly.
GuaranteeProblem guarantees k is valid (1 ≤ k ≤ n), so ans will always be set before traversal ends. No need for a null-check on the return value.

var kthSmallest = function(root, k) {
  let ans = null, count = k;

  const inOrder = (curr) => {
    if (ans != null) return;          // already found → skip
    curr.left  && inOrder(curr.left);  // ① go left first
    if (--count === 0) ans = curr.val;  // ② visit root: kth?
    curr.right && inOrder(curr.right); // ③ go right last
  };

  inOrder(root);
  return ans;
};

⚠️ Gotchas & Common Pitfalls
  • 1-based vs 0-based: The target k is 1-based. You decrement k after popping from the stack.
  • Not Breaking Early: The whole point of iterative Inorder is you stop exactly when you hit the kth element. Do not mistakenly traverse the entire tree!
⚖️ Trade-offs & Alternatives
  • Iterative Inorder (Current): O(H + k) Time, O(H) Space. Halts early. Best choice for one-off queries.
  • Recursive Inorder: O(N) Time, O(H) Space. Without ugly global variables and exception throwing, a standard recursion usually visits the whole tree.
Time: O(H + k) — traverse down to min, then process k nodes
Space: O(H) — stack depth up to tree height
Pattern: Iterative In-order DFS
