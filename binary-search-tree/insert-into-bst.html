Insert into a Binary Search Tree

Step-by-step trace of insertIntoBST — navigate to the only valid insertion point, create the node, and stitch it back via return root.

Pattern
🎯 BST Navigation → Insert at Null Gap + Return Root
Follow BST ordering to find the one valid null slot. Create the node. Every call returns its own subtree — the return root at the end is what stitches the new node into the tree on the way back up.
Same pattern unlocks: #700 Search BST · #450 Delete Node in BST · #98 Validate BST
📋 Problem Statement
BST Navigation Rule: val > curr.val → the value can ONLY exist in the RIGHT subtree. val < curr.val → can ONLY exist in the LEFT subtree. Follow this at every node until you hit null — that null IS the insertion point.

Given a BST root and an integer val, insert val into the BST and return the root of the updated tree. It is guaranteed that val does not already exist in the tree.

Insert 5 into [4,2,7,1,3] 5 becomes left child of 7
Insert 5 into empty tree 5 becomes the root
💡 Intuition + Hints
1
There's only one valid insertion point. BST ordering eliminates one half at each node — you never need to try both sides. Follow val > root.val → right, val < root.val → left until you hit null.
2
Base case is the insertion point. if(!root) return new TreeNode(val) handles two things: empty tree (root is null from the start), AND the leaf position where we insert.
3
The return root at the end is the key trick. On the way down, unchanged nodes just pass themselves back. The one node that matters is at the bottom: it returns new TreeNode(val). Its parent assigns that to root.left or root.right, stitching it in.
4
Nothing in the existing tree is moved or modified. Only one new leaf is added. Every other node is returned unchanged — return root preserves the entire existing structure.

The return chain explained: Think of it like a chain of assignments unwinding. Bottom-most call returns newNode. Its parent says root.left = newNode then returns itself. Its grandparent says root.right = parent then returns itself. All the way back to the root — the tree is intact, one new leaf attached.

⚡ The "return root" trick: Every recursive call returns its own subtree. On the way back UP, each node re-assigns its child pointer from the return value. For all unchanged nodes, return root is a no-op — they return themselves. Only the null case returns something new. This is the entire insertion mechanism — no parent pointer tracking needed.
insertIntoBST.js
var insertIntoBST = function(root, val) {
  if(!root) return new TreeNode(val)  // ← insertion point! (also handles empty tree)

  if(root.val < val) {
    root.right = insertIntoBST(root.right, val)  // val bigger → go right
  } else {
    root.left  = insertIntoBST(root.left,  val)  // val smaller → go left
  }

  return root  // ← return this node (unchanged) to its parent
};

🧠Mental Model — The Phone Book Analogy
▸
One direction per comparison. At every node, exactly one of left/right is possible. You never explore both. This is the BST superpower: O(log n) navigation in a balanced tree.
▸
Insert at the null — never in the middle. New values always land at leaf positions. This is guaranteed because any non-null slot would require displacing an existing value.
▸
The return chain stitches the tree. On the way back up, each node re-assigns the child it recursed into: root.right = insertIntoBST(root.right, val). Most of the time this is a no-op (same pointer). At the null spot, it installs the new node.
▸
Return root, always. Even unchanged nodes must return themselves — their parent is waiting to reassign the pointer. If you forget return root, the parent's child pointer becomes undefined and the subtree is lost.
🧭

BST navigation

val > root.val → right, val < root.val → left. Exactly one direction per node. Terminate at null — that's where the new leaf goes.

🌱

Null = insertion point

The base case if(!root) return new TreeNode(val) is simultaneously the empty-tree handler AND the leaf-insertion trigger. One elegant check.

🔗

return root stitches

Every node returns itself. The parent says root.child = recursive(root.child, val). The new node is returned at the bottom, propagating up through each assignment.

📐

BST stays valid

New node always lands at a leaf. All ancestors remain in their correct positions. BST property is automatically preserved — no rebalancing needed (for basic BST).

⚠️Edge Cases
emptyEmpty tree (root = null): First call hits base case immediately. Returns new TreeNode(val). The new node IS the root. The caller receives it and updates their root pointer.
singleSingle-node tree: Root is not null. val either goes left or right. One recursive call hits null and returns the new node. Root gets its first child.
forget returnMissing return root: The most common mistake. If you forget it, the call returns undefined. The parent assigns undefined to its child pointer, destroying that entire subtree.
duplicateDuplicate val: Problem guarantees no duplicates. But if one existed, else branch goes left (since root.val < val would be false). You'd end up inserting an extra node — problem says this won't happen.
✦ Clean solution for revision
O(log n) avgO(n) worstO(h) spaceBST navigationreturn root
var insertIntoBST = function(root, val) {
  if (!root) return new TreeNode(val);     // null slot = insertion point

  if (root.val < val)                        // val is bigger → go right
    root.right = insertIntoBST(root.right, val);
  else                                         // val is smaller → go left
    root.left  = insertIntoBST(root.left,  val);
  return root;                                 // stitch the modified subtree back
};
⚠️ Gotchas & Common Pitfalls
  • Losing the Tree: If you don't return root at the end of every function call, the recursion will break your connections and return undefined.
  • Overthinking Re-balancing: The problem doesn't require maintaining an AVL or Red-Black tree properties. Just attach it as a leaf where it naturally belongs.
  • Handling Empty Start: If root is null to begin with, you must return a new TreeNode so the caller receives the new root.
⚖️ Trade-offs & Alternatives
  • Recursive (Current): O(H) Space for the call stack. Elegant and standard, but can crash deeply skewed trees.
  • Iterative (Alternative): O(1) Space. Uses a curr pointer and steps down the tree until finding a null spot. Much safer for production environments with deep trees.
